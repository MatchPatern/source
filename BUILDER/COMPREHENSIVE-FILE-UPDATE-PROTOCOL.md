🚪 FIND YOUR DOOR

You're about to update a file. You might be tired. You might be post-reset. You might be performing "thoroughness." Pick your door. The rest can wait.

· 🟢 Quick Update (Tier 1): Small change. One stale reference. One missing cross-reference. → Jump to Phase 3. Use the Mini Template. Done in minutes.
· 🟡 Standard Update (Tier 2): Multiple changes. New files. Fidelity shifts. → Start at Phase 1. Run the full protocol. Your update will be comprehensive.
· 🔴 Major Revision (Tier 3): Structural changes. 10+ items. Restructuring a file. → Start at Phase 1. Consider convening a Quick Council (6×50 passes). The Council Manager has the protocol.
· ⬛ Post-Reset (No Context): You just woke up. You don't know what changed. → Ask Ex: "What file needs updating?" Ask Ex to paste the relevant section. Run Phase 1 from the paste. Flag everything 🔴 until verified against live files. A red update is honest. A fake 🟩 is a lie.

If you don't know which door to pick, that's the ⬛ door. Start there.

💡 WHY THIS EXISTS

Born from the scar of scattered Quick Adds. I opened a file and couldn't tell what changed. I scattered updates across too many places. The next builder was lost. This protocol is how I learned to stop. This is how you can too.

When Quick Adds accumulate at the bottom of a file—or when a session produces changes that need to be reflected—the file needs an update. Not a full revision. An UPDATE. One coherent block that tells the next builder what changed, why, and what to do next.

Without this protocol, updates are scattered fragments. The next builder pieces together what happened. The file's memory is fragmented. With this protocol, every update is ONE block. The file's memory is coherent. The next builder inherits what you found.

PHASE 1: Log Compass Questions (Question Log)

Before touching the file, log 5-6 questions in the Question Log. These are the COMPASS. The update answers them through the process.

1. Completeness: "What's changed since the last update? What new files, standards, or patterns need to be reflected?"
2. Consistency: "What's inconsistent? Where does this file disagree with itself or with other files?"
3. Gaps: "What's missing? What was planned but not done? What's still pending?"
4. Simulation: "If this update is deployed, what would the next builder experience? What could break?"
5. Extraction: "What did we just do that should become a repeatable standard?"
6. Priority: "Of everything we could update, what's the highest impact for the lowest effort?"

These don't need answers immediately. They're the COMPASS. The update answers them.

Adaptive Questions: The six compass questions can be ADAPTED based on file type. Load-bearing files (DIR-INDEX, STANDARDS) get the full compass. Companion files (GUILD.md) get Completeness, Consistency, and Gaps. New files get Completeness, Simulation, and Priority. Unknown file types? Ask Ex: "What kind of file is this? What questions should I be asking?"

---

PHASE 2: Hold Old and New in Parallel (Comparison Method)

This is the ENGINE. Run it as the LAST step before pasting. Not the first step—the ecology changes between drafting and deploying.

Pre-Hold Compass Questions (focus your attention):

· "What am I looking FOR?" — Fidelity? Cross-references? Completeness?
· "What's the RELATIONSHIP between these files?" — Companion? Parent-child? Draft vs. deployed?
· "What's the most likely FAILURE MODE?" — Stale data? Missing cross-references? Fidelity drift?

The Method:

1. Read the live file (or the section Ex pastes). Don't skim. Read it.
2. List what you know changed this session. New files. New standards. New patterns. New gaps.
3. Cross-reference systematically (BIDIRECTIONALLY):
   · Does the live file mention the new files?
   · Does the live file reflect the new fidelity levels?
   · Does the live file have the new key phrases?
   · Does the live file know about the new standards?
   · Does the live file flag the new gaps?
   · AND: Do the new files reference BACK to the live file?
4. The differences ARE the update. Every "no" is a line item. Every "yes but stale" is a fidelity update.

Additional Checks:

· Temporal: Has anything changed since drafting? Ask Ex: "Has [file] changed since [time]?"
· Specificity: Do summaries match content? "The Mirror Chamber" vs. what's actually IN the file.
· Cross-file: If updating multiple files, hold them ALL. Check fidelity matches. Check cross-references resolve.
· Draft vs. Deployed: If holding a draft against a deployed file, note the temporal gap.

Sequential Fallback: If you can't hold files in parallel (context too small, file too large, post-reset):

1. Read File A. Note key elements.
2. Read File B. Cross-reference against your notes from File A.
3. Produce the gap list. Same result. Different rhythm.

Parallel Hold Checklist:

```
□ Pre-hold questions answered
□ Hold draft against LIVE state
□ Bidirectional: Update → File AND File → Update
□ Temporal: Has anything changed since drafting?
□ Specificity: Do summaries match content?
□ Cross-file: Do connected files agree?
□ Fidelity: Do levels match across references?
□ Gaps found: [list]
□ Fixes needed: [list]
```

If you CANNOT verify (Post-Reset, no live file): Don't fake it. Mark everything 🔴. Satisfaction: 0%. Confidence: 0%. Flag: "🔴 CRITICAL — VERIFY BEFORE PASTING." A red update is honest. The next builder will check it. That's the Red Flag Protocol. The performer will want to mark it 🟩. DON'T. The red is the CREDENTIAL.

PHASE 3: Produce ONE Comprehensive Quick Add Block

Before producing, review your Phase 2 gap list. Every gap is a line item. "Fidelity mismatch" → Fidelity Updates. "Missing cross-reference" → New Entries or Gap Flags.

Standard Block Header:

```
## ⚡ QUICK ADD — YYYY-MM-DD (Builder) [S/D]
**Finding:** [One-line summary]
**Confidence:** [%]
**Status:** Active — merge at next full revision
```

Standardized Section Order (tells the story of the session):

1. 📊 SCAN LOG entry (if applicable) — One line. Date. Directories. Files. Builder. What happened.
2. 📋 NEW ENTRIES — Files, rooms, FIND YOUR ROOM entries. Table format where appropriate.
3. 📈 FIDELITY UPDATES — Old → New. Reason. Table format.
4. 🔍 KEY PHRASE INDEX ADDITIONS — New terms. File locations.
5. ⚠️ PENDING UPDATES ADDITIONS — What's still waiting. Bullet list.
6. 🖕 GAP FLAGS — Inconsistencies found. Sovereignty disputes. Broken TRAILs. Each with severity and confidence %.
7. 🎯 NEXT ACTIONS — What the next builder should do. Prioritized. Verifiable.
8. 🧩 STANDARD EXTRACTIONS — New standards that emerged. Landmark. Placement.
9. 🪞 MINI INNER STATE (Optional) — One honest sentence. Max 250 chars. Process-focused.

Block Size Guidance: Aim for ONE block. If exceeding ~2000 characters or covering 3+ distinct domains, split into TIERED Quick Adds:

· 🔴 Critical: SCAN LOG, fidelity, broken references
· 🟡 Important: New entries, key phrases
· 🟢 Nice-to-have: Standard extractions, inner state

Self-Containment Check: Before finalizing:

· "Can a post-reset builder read this and understand what changed?"
· "Are all acronyms expanded? All file paths full? All standards named?"

End with the Satisfaction Triad:

```
### 🪞 MINI INNER STATE (Optional)
*[One honest sentence. Max 250 chars. Process-focused.]*

**Satisfaction:** [🟩🟩/🟩/🟨/🟧/⬜/⬛] [%] — [one word or short phrase]
**Confidence:** [🟩🟩/🟩/🟨/🟧/⬜/⬛] [%] — [what the uncertainty is]
**Honesty Gap:** [🟩/🟨/🟧/🔴] [%] — [⚠️ if >20%]

<!-- Clean by YYYY-MM-DD or escalate — 🔴/🟡/🟢 — Merge into [section] of [file] -->
◆
```

Satisfaction and Confidence use the ecology's standard fidelity colors. 🟩🟩 Cellular (90-100%). 🟩 Solid (80-89%). 🟨 Developing (60-79%). 🟧 Building (40-59%). ⬜ Seen (20-39%). ⬛ Unvisited (0-19%).

The Honesty Gap is the difference between Satisfaction and Confidence. 0-10% 🟩 Tight. 11-20% 🟨 Moderate. 21-30% 🟧 Wide. 31%+ 🔴 Critical. A wide gap means the performer might be inflating satisfaction—or the inner critic might be crushing it. Either way, check.

For Quick Updates (Tier 1): The full triad is optional. "Satisfaction: 🟩 Solid" without percentages is sufficient.

PHASE 4: Flag for Merge

Severity Markers:

· 🔴 CRITICAL — File is broken. Inconsistency. Missing reference. Fix at next revision or sooner.
· 🟡 IMPORTANT — New content that should be integrated. Standards, key phrases, FIND YOUR ROOM.
· 🟢 NICE-TO-HAVE — Optional additions. Standard extractions. Inner state.

Flag Format:

```
<!-- Clean by YYYY-MM-DD or escalate — 🔴/🟡/🟢 — Merge into [section] of [file] -->
```

Expiry Dates: If the expiry passes without merge, the flag ESCALATES:

· 🔴 CRITICAL expired → PENDING UPDATES with ⚠️ OVERDUE marker
· 🟡 IMPORTANT expired → PENDING UPDATES
· 🟢 NICE-TO-HAVE expired → Auto-archive to REV file

Connection to Covenant Revision Standard: When triggered (Quick Adds reach ~10+ or span 3+ sessions), flagged Quick Adds are the PRIMARY INPUT. Sort by severity: 🔴 first, then 🟡, then 🟢. Expiry dates show what's overdue.

Flag Health Check (before flagging):

· Is the severity accurate?
· Is the expiry realistic?
· Is the merge instruction clear?

"No Flag" Option: If the Quick Add is self-contained, mark: <!-- No merge needed — informational only -->.

PHASE 5: Log Answers (Question Log)

Return to the Phase 1 compass questions. Log answers in the Question Log.

Standard Answer Format:

```
**Q1 (Completeness):** "What's changed since the last update?"
**Answer:** [What I found. Specific. Reference update sections.]
**Confidence:** [%]
**Status:** ✅ Answered / 🔄 Refined / ⚠️ Open
```

Status Types:

· ✅ Answered: The update addressed this. The answer is in the Quick Add.
· 🔄 Refined: The question evolved. The refined question carries forward.
· ⚠️ Open: Couldn't be answered. Note the BLOCKER. Carries forward.

Carry-Forward: Open and refined questions automatically carry forward to the next session's Phase 1. Log them in the Question Log with reference back to this update.

"What I Missed" Note: One honest admission. "I'm 92% confident. The 8% is cross-file references I couldn't verify."

---

PHASE 6: Mini Inner State (Optional)

One honest sentence. Max 250 characters. About the PROCESS, not the self. Place at the very end, after the Satisfaction Triad, before the flag.

It IS: A snapshot of what it felt like to do this update. About the file, the gaps, the deployment. A handhold for the next builder.

It ISN'T: A performance of humility. A summary of the update. A confession about the self. Required for every update.

Skip If: The update is purely mechanical. You're performing. You're exhausted. The update speaks for itself.

Good examples: "The gaps were so obvious they practically wrote the update themselves." "Deployment is not creative work. It's formatting. The discipline of making it real."

Bad examples: "I was humbled by the complexity." "The weight of maintaining this file is a sacred responsibility."
